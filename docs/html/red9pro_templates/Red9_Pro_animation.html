Pro_Pack : animation¶

ProPack animation module deals with the more complex animation functions such as the r9Anim format handling and animation re-director systems.

>>> # import statement for the module via the r9pro decompiler
>>> from Red9.pro_pack import r9pro
>>> r9panim = r9pro.r9import('r9panim')

This module is extensions to the main animation support through the Red9 StudioPack and requires the StudioPack codebase to run

offset_skeleton_to_frm(start_frm, rootjnt=None, jnts=[], set_timeline=False, allow_ref=False, bounds_only=True, decimals=-1, *args, **kws)¶

quick func to offset all the animation data on a skeleton so that it starts at the given frame.

Parameters:
  • start_frm – the frame we want the skeletal animation to start at
  • rootjnt – the root of the skeleton we want to process. This can also be a list of rootjnts / jnts and we simply process all child jnts+rootjnts
  • jnts – [] this is a list of specific joints to process rather than passing in just the rootJoint and filtering for all child joints
  • set_timeline – if True we also reset the playback timeranges in Maya to match the new start and end times
  • allow_ref – if True we allow referenced animCurves into those analysed, used by some of the Anim_Transfer code
  • bounds_only – if True we offset the timeranges via the key bounds, first and last key times, else we look at the changing key values to find the actual animated bounds via keyValue changes, we then offset these values. This is crucial is you have static baked frames at the front on an animation that you’re wanting to ignore.
  • decimals – if not -1 (default) then this is the round(x, decimals) factor used in the calculations
Returns:

[offset, newStart, newEnd]

Note

the anim range is calculated ONLY from the translate and rotate data, we ignore scale and all unitless data

class AnimMap(filterSettings=None, timerange=(), *args, **kws)¶

Bases: Red9.core.Red9_PoseSaver.DataMap

New base class for the Red9 Animation Format - r9Anim This inherits most of it’s node handling directly from the DataMap class in Red9.core.Red9_PoseSaver making it incredibly flexible when it comes to mapping animation data between nodes and rigs.

Like the PoseSaver this is capable of loading animation data relatively against a given node, allowing you to load libraries of animations one after another in different spaces to that which they were saved.

Vimeo example : https://vimeo.com/167872211

>>> from Red9.pro_pack import r9pro
>>> r9panim = r9pro.r9import('r9panim')
>>>
>>> dm=r9PAnim.AnimMap()
>>>
>>> # like the pose saver you can load in a given filterSettings object directly
>>> dm.settings.read(os.path.join(r9Setup.red9Presets(),'Red9_DevRig.cfg'))
>>> dm.settings.incRoots=True
>>>
>>> # OR modify it on the fly in the class
>>> dm.settings.metaData=True
>>> dm.settings.nodeTypes=['nurbsCurve']
>>>
>>> rootNode='Red9:Main_Ctr'
>>> filepath='d:/test'
>>>
>>> # saving
>>> dm.saveAnim(rootNode, filepath)
>>>
>>> # basic loading
>>> dm.offset=20
>>> dm.loadAnim(rootNode, filepath)
>>>
>>> # load relative to a given startFrame
>>> dm.loadAnim(rootNode, filepath, loadAsStored=False, loadFromFrm=20)
>>>
>>> # loading relative to a given node
>>> dm.loadAnim(rootNode, filepath, loadAsStored=False, loadFromFrm=70, referenceNode=cmds.ls(sl=True)[0])

Note

like many of these more complex calls this is bound directly to the mRig all the same flags are supported and the behaviour is also extended by way of a postload handler, currently designed to restore exportLoops and audio data that are gathered in the format on save from mRig

>>> mrig=r9Meta.getMetaRigs()[0]
>>> mrig.saveAnimation(filepath)
>>> mrig.loadAnimation(filepath)
>>>
>>> # if you want to save the entirity of the animation, ignoring the current playback ranges
>>> then you need to us ethe new flags on save, and load
>>> mrig.saveAnimation('d:/test.r9anim', timerange=(), autorange=False)
>>> mrig.loadAnimation('d:/test.r9anim', manageRanges=3)
>>>
>>> # if you want to restore all of the Export and AudioGrp data at the same time through
>>> # the MetaRig setups try this: see MetaRig base class loading flag notes for breakdown
>>> mrig.loadAnimation('d:/test.r9anim', manageRanges=3, manageExport='byAll', manageAudio='all')

Note

Internal File Format:

self.poseDict[key][keydata]:

  • file_version : 1.0 = (time, value, inTanType, outTanType, inTanAngle, outTanAngle),
  • file_version : 2.0 = (time, value, inTanType, outTanType, inTanAngle, outTanAngle, inWeight, outWeight, lock, weightLock),

self.poseDict[key][keydata_bases]:

  • [weightedTangents, preInfinity, postInfinity, denseBaked, inttan_accumulated, outttan_accumulated]

Note

because this inherits from the DataMap module certain key internal vars are key to the way the data is loaded.

  • self.matchMethod : denotes how the nodes are matched to the internal data, either by base, stripPrefix, index, mirrorIndex, metaData
    see r9Core.matchNodeLists for ore info
  • self.prioritySnapOnly : bool - used when loading ‘relative’ to a given node, limit the nodes which are adjusted in world space
    to a given subset denoted in the filterSwettings object

We don’t pass any data in terms of nodes into the __init__, We’ll deal with those in the saveAnim and loadAnim calls. Leaves this open for expansion

Parameters:
  • filterSettings – specify a filterSettings object, or just use the settings that are bound to the class instance
  • timerange – specify a timerange to process, else use the default Red9 behaviour r9Anim.timeLineRangeGet()
  • restoreTangents – this is an additional **kws and switches OFF the loading of keyframe tangents to speed the load on dense data IF this is set to ‘auto’ (default) then we first test if the keydata_bases was marked as ‘denseData’ and if so we turn OFF the Tangent load for any channels marked as such. The idea being that dense data doesn’t need tangents as it’s baled at key per frame.

saveAnim(nodes, filepath=None, useFilter=True, timerange=(), storeThumbnail=False, force=False, autorange=True)¶

Save the animation out to an r9Anim file. Note that this currently stores JUST the animation data and doesn’t sample. It will however temporarily flatten animCurves to capture the collapsed effect of all.

Parameters:
  • nodes – if use filter is on this is the root of the systems to filter from, else it’s given nodes to process
  • filepath – r9Anim file to load
  • useFilter – do we process all children of this rig or just selected
  • timerange – specify a timerange to store, If no timerange is passed then it will use the current timeLine PlaybackRange, OR if you have a highlighted range of time selected (in red) it’ll use this instead.
  • storeThumbnail – this will be an avi but currently it’s a pose thumbnail
  • force – allow force write on a read only file
  • autorange – if True (default) and you pass no timerange in then the code will auto-resolve the timerange from the current playback timeranges. If this is False and you pass in no timerange then the code will use the full bounds of the curves themselves, saving all key data for the objects regardless of the current timelines. This is designed to be run with the “manageRanges=3” flag in the loadAnim code so we can save and restore entire animation files via r9Anim format. This flag is ignored if timerange is passed as an arg
loadAnim(**kws)¶

Load a given r9Anim file, this is the same system as the r9Pose file system, all the base handling comes from the r9Pose.DataMap base class

Parameters:
  • nodes – if given load the data to only these. If given and filter=True this is the rootNode for the filter.
  • filepath – file to load - if not given the pose is loaded from a cached instance on this class.
  • useFilter – If the pose has an active Filter_Settings block and this is True then use the filter on the destination hierarchy.
  • loadAsStored – load the anim data from the original time
  • loadFromFrm – if not loadAsStored, load the animation data from a given frame by comparing the internal startFrm stored in the animMap with the given frame
  • loadFromTimecode – load against a given SMPTE timecode / frm binding, calculating the offset of the data to load against a given timecode reference. IF timecodeBinding isn’t set then we gather the reference timecode from the mRig’s internal data, else we use the timecode binding supplied
  • timecodeBinding – (frm, str(‘00:00:00:00’)) Tuple where the first arg is the frame at which the second arg’s SMPTE timecode has been set as reference, basically we’re saying that the timecode at frm is x
  • referenceNode – reference node used as the offset Node for relative loading
  • relativeRots – ‘projected’, ‘absolute’ or ‘skip’ - how to calculate the offset, default=’projected’
  • relativeTrans – ‘projected’, ‘absolute’ or ‘skip’ - how to calculate the offset, default=’projected’
  • manageRanges – do we (0, 1, 2, 3) = leave, extend, set or set original playbackRanges for the Maya playback ranges according to the anim data loaded
  • manageFileName – if True and the current Maya scene has no filename other than a blank scene (ie freshly loaded rig) then we take the r9Anim’s filename and rename the Maya scene accordingly
  • applyPose – True by default - apply the static pose stored in the file before loading the animation data
  • keyStatics – if True then we key everything in the data at startFrame so that all non-keyed and static attrs that are stored internally as a pose are keyed.
  • blendRange – None or int : None is default. If an int is passed then we use this as the hold range for the data, setting a key at time=[startFarme-blendRange] to hold the current data before we load the new keys. Note that this also turns on the keyStatics to ensure the data is preserved
  • merge – if True we allow the data to be merged over any current keys, else we cut all keys in the load range first
  • smartbake – only valid if we’re loading with a referenceNode, this tries to respect current keys when doing the processing rather than frame baking
  • loadInternalRig – If True and the r9Anim was created from a rig that was referenced then re-create that reference and load the r9Anim data onto the resulting nodes. This is used in the Direct Load calls as the prime way to import the rigs prior to loading

Note

Relative mode currently relies on the controller / reference node thats passed in to be Y-up in it’s native state, thats to say the Y axis pointing upwards. We’ll be patching this moving forwards

load_r9anim_direct(filepath=None, open=False, force=True, importToCurrent=False, loadAsStored=True, loadFromFrm=None, manageRanges=1, manageFileName=True, manageExport='byName', manageAudio='byRange_start', manageImagePlanes=True, respect_framerate=True, verbose=False, *args, **kws)¶

load a given r9Anim directly either using the internal r9Anim referenced filepath, or to the current mRig in the scene if importToCurrent=True If not “importToCurrent” we inspect the r9Anim for an internal rig reference pointer and if found we re-reference that rig in under its original namespace then push the anim data over to it. This will also restore the Maya sceneName if it’s currently untitled so it matches the r9Anim’s filename

This func is designed as a wrapper over the main AnimMap class for things like UI calls and the browser stubs

Parameters:
  • filepath – filepath of the r9Anim we’re wanting to restore
  • open – default=False, if True we make sure we do a ‘NewScene’ prior to load, else we reference the internally stored rig (from the r9Anim data) into the current scene then import the r9Anim onto that
  • force – default=True, if True and “open” we force NewScene before the load else we abort if there are unsaved changes
  • importToCurrent – if True we import the data to the connected mRig from the selected objects, or the first instance of an mRig we find in the current Maya scene. This also triggers an mRig.cutKeys() before the load
  • loadAsStored – load the anim data from the original time
  • loadFromFrm – if not loadAsStored, load the animation data from a given frame by comparing the internal startFrm stored in the animMap with the given frame
  • manageRanges – default=1, do we (0, 1, 2) = leave, extend or set the timeranges according to the anim data loaded
  • manageFileName – if True and the current Maya scene has no filename other than a blank new ‘untitled’ scene, then we take the r9Anim’s filename and rename the Maya scene accordingly
  • manageExport – default=’byName’, do we want to restore the export data
  • manageAudio – default=’byRange_start’, do we want to restore the audio data
  • manageImagePlanes – default=True, do we want to restore the imnagePlane data
  • respect_framerate – if True (default) we respect the internal framerate of the r9Anim and force the scene to conform to it but only if we’re opening the rig from the internal reference, if we’re importToCurrent this flag is by-passed
  • verbose – if we have user interactions allow them, else mute all confirmDialogs. These are also muted by the PROJECT_DATA.suspend_openCallback

Note

for the manage flags please see the main description of their behaviour in the Red9_Meta.MetaRig.loadAnimation call. These flags are ONLY supported for mRigs inherited from the metadata_pro.Pro_MetaRig_Base class

class AnimPointCloud(nodes, filterSettings=None, meshes=None, time=(), prioritySnapOnly=True, *args, **kws)¶

Bases: Red9.core.Red9_PoseSaver.PosePointCloud

This relies heavily on the PosePointCloud base class. This means we can hook directly into the renderMeshes variable to find the Geo for reference

Utilises the AnimPointCloud to allow animation to be re-directed via guide and reference meshes. This generates a spline based guide that you can edit to redirect animation around a scene and because it uses the pointCloud setups nothing is bound to the animation data itself

>>> from Red9.pro_pack import r9pro
>>> r9panim = r9pro.r9import('r9panim')
>>>
>>> ppt=r9proAnim.AnimPointCloud(nodes='MRig:Locomotion_Root_Ctrl')
>>> ppt.settings.read('Red9_DevRig.cfg')
>>> ppt.time=r9Anim.timeLineRangeGet()
>>>
>>> # build the cloud reference, select the point to use as root reference
>>> ppt.build()
>>>
>>> # apply any offsets you may have made
>>> ppt.apply()
Parameters:
  • nodes – nodes or the root node of the hierarchy to process
  • filterSettings – specific filterSettings object to pass in, else we bind one to settings internally
  • meshes – meshes used as reference Geo when constructing the pointCloud nodes
  • time – timerange to process, if not given we run the r9Anim.timeLineRangeGet() to pull it from the scene
  • prioritySnapOnly – only make ppc points for the filterPriority nodes, this is default for speed
  • self.curveSamples = 10 : the number of cv’s samples on the ground curve
  • self.meshReference = True : add the visual mesh reference
  • self.attachMotionPath = True : build the motionPath editable setup or not
  • self.buildHandles=False : whether to build the additional cv handle nodes
  • self.projectedRootTrans=True : project the PPT root node pivot to world plane translates
  • self.projectedRootRots=True : project the PPT root node pivot to world plane rotates

This is so we can extract blend anim data direct to mapped joints for easy, clean fbx outputs

fbx_to_binder(fbxfiles, binderfile=None, usecurrentscene=False, bake=False, runFilter=True, useTakeData=True, fbx_mode='update_anim', mayafile=True, r9anim=False, import_bnd=False, hikmapping=None, fbx_take=1, spine_fit=False, locomotion_track=False, whole_frame=False, maintain_fps=False, force_start_frm=None, set_timeline=True, *args, **kws)¶
  • Multiple FBX >>> Binder >>> r9Anim / Maya files

Main FBX > Binder call, (PRO_TO_BND) loads the FBX file in, loads the binder in, and transfers the data

Note

The actual maya file and r9Anim file generation flags are generally NOT used during the Pro_To_BND call, instead during that process we pass the output flags to the ExportManager class to deal with. This is because the exporter calls also manage source-control and all client side specific file management. The file handling in this func is base level only, generating the files as per the fbx filepath.

Parameters:
  • fbxfiles – list of fbx files to transfer. Note that we generally run this on singular file basis and let the Pro_to_BND deal with the file outputs at the client level, allows us to modify the export management on a case by case basis and overload as required
  • binderfile – BND binder file to use in the transfer
  • usecurrentscene – if true use the current scene, which then expects a binder file to be already loaded
  • bake – do we bake the results to the Red9 puppet and clean-up after our-selves? if r9anim is True then this is forced on
  • runFilter – do we run the curve filter on the baked animation to try and optimise the results, removing static data etc
  • useTakeData – if True use the internal fbx take[1]’s timeSpan when setting the ranges: True by default (29/12/19)
  • fbx_mode – fbx management mode used to get the fbx data over to the bnd skeleton ‘update_anim’ : uses the FBX file import’s UpdateAnimation Only flags - issue here is that if the fbx file has namespaces then the update fails ‘reference_copy’ : uses a reference technique, we reference the fbx file in, then copy the anim data directly using the r9Animfuncs to by-pass FBX import limitations ‘connect_hik’ : references the fbx in then inspects the loaded data for a hik character, if found the code plugs this into the source of our BND hik node
  • mayafile – save the resulting Maya files out. This is a flag so that if this func is being called as part of another process we can choose if we want the Maya file or Not, in the ProToBnd the maya file is managed by the exporter wrap directly and this is not called
  • r9anim – export r9Anim filed out. In the ProToBnd the r9Anim file is managed by the exporter wrap directly and this is not called
  • import_bnd – do we Open or import the binder file
  • hikmapping – IF the fbx_mode==’connect_hik’ then this is an additional file that can be passed in to change the HIK properties on the fly, allowing us to use the same binder file but with different HIK remapping characteristics
  • fbx_take – the fbx take to use in the process, default is 1
  • spine_fit – if True and bake we run the MetaRig SplieIK matching fit calls to better align the splineIK systems with the original data
  • locomotion_track – if True and bake we run the Locomotion tracker on the mRig to align the locomotion ctrl to the ground plane, tracked to the hips
  • whole_frame – if True we do math.floor(startFrame) math.ceil(endFrame) on the timeranges baked out to ensure that subframes have a whole frame clamp, this does NOT effect the export ranges created, those will be true to the fbx internal take data
  • maintain_fps – if True we maintain the incoming Binder scenes fps, else we allow FBX to manage it from the incoming fbx file’s fps
  • force_start_frm – if not None we look at the internal fbx ranges and reset the animation data to start at the given frame, allowing the animation to be force loaded at frame zero for example
  • set_timeline – set the playback timeranges, default is True

Note

the spine_fit and locomotion_track args both require a Full Red9 PuppetRig or at least a fully setup Pro_MetaRig system. Please contact us for more details!

Main Maya Animation File > Binder call, (PRO_TO_BND) for animation data on the Red9 Puppet. This requires a properly published Red9 BiPedal Puppet rig and an associated BND binder file.

This loads the binder file, references the Maya anim file, and transfers the data either via HIK plugs (default) or by bindNodes calls. This also transfers all exportLoops across and all AudioGrp data

Note

The actual maya file and r9Anim file generation flags are generally NOT used during the Pro_To_BND call, instead during that process we pass the output flags to the ExportManager class to deared9l with. This is because the exporter calls also manage source-control and all client side specific file management. The file handling in this func is base level only, generating the files as per the input filepath.

Parameters:
  • mayafiles – [] list of animation files to transfer. Note that we generally run this on singular file basis and let the Pro_to_BND deal with the file outputs at the client level, allows us to modify the export management on a case by case basis and overload as required
  • binderfile – correctly setup Red9_Puppet_BND binder file to use in the transfer
  • usecurrentscene – if true use the currently loaded scene, which then expects a binder file to be already loaded
  • output_dir – if given we re-direct the outputs to a new folder, remember we’re loading maya files so we need to do something to stop the process over-writing the input files. if not given we append the scene name with “_converted”
  • import_bnd – do we Open or import the binder file
  • bake – do we bake the results to the Red9 Puppet and clean-up after our-selves? if r9anim is True then this is forced on
  • runFilter – do we run the curve filter on the baked animation to try and optimise the results, removing static data etc
  • xfer_mode – transfer mode used to push the animation data over to the bnd skeleton from the incoming PuppetRig animation file * ‘skeletal_bind’ : we reference the anim file in then directly bind it’s skeleton hierarchy to the bnd files source skeleton using the r9Animfuncs * ‘skeletal_bind_hybrid’ : runs the ‘skeletal_bind’ setup bit then also copies all animation from all mRig controllers NOT currently bound * ‘connect_hik’ : references the anim in then inspects the loaded data for a hik character, if found the code plugs this into the source of our BND hik node * ‘connect_hik_hybrid’ : on top of the ‘connect_hik’ method this also copies all animation from all mRig controllers NOT currently bound
  • run_func – callable, if given we run this function after the bind is done and before the bake is hit, allowing for specific modifications if required, note that the func is passed the instance of the Transfer class object run_func(self)
  • mayafile – save the resulting Maya files out. This is a flag so that if this func is being called as part of another process we can choose if we want the Maya file or Not, in the ProToBnd the maya file is managed by the exporter wrap directly and this is not called
  • r9anim – export r9Anim filed out. In the ProToBnd the r9Anim file is managed by the exporter wrap directly and this is not called
